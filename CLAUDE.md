# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Repository Is

This is the **ai-specs framework** — a portable, specs-driven development framework that provides AI agent definitions, slash commands, and development standards. It is designed to be copied into any project to enable consistent AI-assisted development workflows. It is NOT an application codebase — there is no build, lint, or test pipeline here.

## Architecture

The framework uses a **symlink-based distribution model**:

- `ai-specs/` is the single source of truth for all content (specs, agents, commands)
- `.claude/commands/` and `.claude/agents/` contain symlinks into `ai-specs/.commands/` and `ai-specs/.agents/`
- `.agent/` (Antigravity) and `.cursor/rules/` (Cursor) reference the same core rules
- Root-level `AGENTS.md`, `GEMINI.md`, and `codex.md` are symlinks to `ai-specs/specs/base-standards.mdc` (this file is the exception — it is standalone)

### Directory Layout

```
ai-specs/
├── specs/                  # Development standards (.mdc files)
│   ├── base-standards.mdc  # Core principles, loaded by all AI tools
│   ├── backend-standards.mdc
│   ├── frontend-standards.mdc
│   └── documentation-standards.mdc
├── .agents/                # AI agent role definitions
├── .commands/              # Slash command templates (source of truth)
└── changes/                # Generated implementation plans per ticket
```

### How the Pieces Connect

1. **Standards** (`specs/*.mdc`) define rules and conventions — agents and commands reference these
2. **Agents** (`.agents/*.md`) define personas with specific expertise that commands can invoke
3. **Commands** (`.commands/*.md`) are workflow templates that orchestrate the Linear-to-PR pipeline
4. **Changes** (`changes/`) hold per-ticket implementation plans generated by planning commands

### Workflow Pipeline

Commands follow a Linear ticket lifecycle: `enrich-us` (enriches ticket) → `plan-*-ticket` (generates plan in `changes/`) → `develop-*` (implements via TDD, opens PR) → `ship-it` (commit + push + PR)

Most commands depend on **Linear MCP** to fetch/update tickets. Plans are saved as `changes/TICKET-ID_backend.md` or `changes/TICKET-ID_frontend.md`.

### Available Slash Commands

| Command | Purpose |
|---|---|
| `/enrich-us [id]` | Enrich a Linear ticket with technical detail |
| `/plan-backend-ticket [id]` | Generate backend implementation plan |
| `/plan-frontend-ticket [id]` | Generate frontend implementation plan |
| `/develop-backend [plan.md]` | Implement backend from plan (TDD) |
| `/develop-frontend [plan.md]` | Implement frontend from plan |
| `/ship-it [context]` | Analyze, document, commit, push, and open PR |
| `/write-tests [file/module]` | Generate comprehensive test suite |
| `/review-architecture [module]` | Architectural compliance report |
| `/setup-mkdocs` | Scaffold MkDocs documentation site |
| `/create-adr [description]` | Create Architecture Decision Record |
| `/update-docs` | Update technical docs from recent changes |
| `/meta-prompt [prompt]` | Improve a prompt using best practices |

## No Build/Test Pipeline

This is a framework of markdown/mdc specs, not an application. There are no build, lint, or test commands. Validation is done by reviewing the markdown content and verifying symlinks resolve correctly.

## Core Principles

- **Small tasks, one at a time**: Work in baby steps. Prefer focused, incremental changes.
- **Test-Driven Development**: Start with failing tests for any new functionality (red-green-refactor).
- **Type Safety**: All code must be fully typed. No `any` in TypeScript.
- **Question Assumptions**: Always question assumptions and inferences.
- **Pattern Detection**: Detect and highlight repeated code patterns.
- **English Only**: All technical artifacts (code, comments, docs, commits, tickets, configs) must be in English.
- **Conventional Commits**: Use `feat:`, `fix:`, `docs:`, `test:`, `refactor:` prefixes.

## Important: Editing Symlinked Files

Always edit the **source files** in `ai-specs/`, never the symlink targets directly. The symlinks in `.claude/commands/`, `.claude/agents/`, and root-level `AGENTS.md`/`GEMINI.md`/`codex.md` all point back to `ai-specs/`. Editing the source ensures all AI tools stay in sync.

## Extending the Framework

**New command:**
```bash
# 1. Create source file
vi ai-specs/.commands/my-command.md
# 2. Symlink for Claude Code
ln -s ../../ai-specs/.commands/my-command.md .claude/commands/my-command.md
# 3. Add Antigravity workflow (if needed)
cp ai-specs/.commands/my-command.md .agent/workflows/my-command.md
```

**New agent:**
```bash
vi ai-specs/.agents/my-agent.md
ln -s ../../ai-specs/.agents/my-agent.md .claude/agents/my-agent.md
```

**New standard:** Create `.mdc` files in `ai-specs/specs/` and reference them from `base-standards.mdc`.
